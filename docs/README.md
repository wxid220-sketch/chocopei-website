# 网站交接总入口

本目录不是“已经完全可上线的成品包”，而是一个包含主开发源码、参考站、原始素材、历史交付与需求文档的工作区。  
后续开发者应先理解目录分工，再决定修复、补件和上线顺序。

## 1. 先说结论

- 当前主开发目录是 `chocopei-website/`
- 参考站是 `参考-zxwlat/`
- 原始素材库是 `可用素材/`
- 历史交付/恢复来源是 `ZXW.LAT_网站成品_交付包.zip`
- 需求来源文档是 `修改完善.docx`
- 当前文档集在 `docs/`

当前状态可以概括为：

- 已经有一套可运行的前端目录骨架；
- 但主源码存在配置损坏、资源路径不一致、品牌残留、文档与真实结构不符等问题；
- 更适合被视为“待整理与修复的开发工作区”，而不是“直接部署即可验收的最终站点”。

## 2. 顶级目录与文件职责

### `README.md`

根目录总入口。  
作用是帮助接手人快速判断每个顶级文件夹/文件的用途、项目当前状态、主要风险和接手顺序。

### `docs/`

本次补充的交接文档目录。  
建议后续所有项目说明、审计结论、实施路线图都持续沉淀在这里，而不是继续把关键信息散落在聊天记录或零散文件里。

### `chocopei-website/`

当前主开发目录。  
这里放的是目前最应该继续修复和演进的站点源码，包括：

- 页面入口：`index.html`、`404.html`
- 部署相关：`sw.js`、`manifest.json`、`favicon.svg`、`CNAME`、`.nojekyll`
- 样式：`css/style.css`、`css/responsive.css`
- 脚本：`js/about.js`、`js/admin.js`、`js/app.js`、`js/audio.js`、`js/cloud.js`、`js/config.js`、`js/earth.js`、`js/effects.js`、`js/guestbook.js`、`js/particles.js`、`js/passion.js`、`js/starmap.js`、`js/utils.js`
- 现有资源：`assets/audio/`、`assets/photos/`、`assets/thumbnails/`

注意：

- 此目录是主战场，但不是“健康状态”；
- 已知 `js/config.js` 有明显损坏和拼接痕迹；
- 站内仍残留 `ZXW.LAT` 品牌文本；
- 资源命名和引用关系与真实目录并不完全一致；
- 目录内还存在一个命名异常的 `js$f` 文件，建议核对是否为误留副本。

### `参考-zxwlat/`

完整参考站。  
这是当前最重要的结构与交互参考来源，可用于：

- 对照页面结构和模块拆分；
- 核对参考站有哪些资源与文件；
- 从完整版本回看某些逻辑或视觉实现；
- 找回主开发目录缺失的文件、路径写法或部署文件。

它更像“对照样本”，不是当前要直接修改后上线的主目录。

### `可用素材/`

原始素材库。  
这里存放的是尚未统一整理进站点资源结构的素材，包括：

- `动画质感人像合集/`
- `回忆录/`
- `热爱/`
- `登录动画和素材/`
- `音乐/`

这些素材可以作为补图、补视频、补音频的来源，但在正式接入前应先做命名、压缩、尺寸与版权/使用范围确认。

### `ZXW.LAT_网站成品_交付包.zip`

历史交付/恢复来源。  
当 `chocopei-website/` 中出现缺文件、路径断裂、部署文件缺失时，这个包可作为恢复和补件来源之一。  
建议把它当作“历史快照”，用于比对和回补，而不是直接覆盖现有主源码。

### `修改完善.docx`

需求来源文档。  
当前很多方向性描述、页面想法和目标诉求，应该以这份文档为准，再结合现有源码和参考站做落地判断。

### `git_history.log`

历史 Git 操作记录。  
该文件包含敏感信息泄露风险，已知存在 GitHub Token 明文，不应继续外发、共享或提交到任何仓库。  
处理建议：

- 立即视为敏感文件；
- 不再传播；
- 尽快轮换相关凭证；
- 后续在仓库忽略规则中排除。

### `user_pasted_clipboard_long_content_as_file_empty.txt`

辅助信息导出文件。  
从当前内容看，里面至少包含历史链接与背景信息片段，例如 TikTok 账号和旧站地址。  
它不是正式需求文档，也不应作为唯一事实来源，可保留作线索参考。

## 3. 当前高风险问题

### 阻塞继续开发的问题

- `chocopei-website/js/config.js` 已损坏，配置可信度不足
- 主源码资源引用与真实 `assets` 目录不完全一致
- 旧版 README 曾把不存在的文件写成“已交付”，容易误导接手人

### 影响上线质量的问题

- `404.html` 仍带 `ZXW.LAT` 品牌
- `css/style.css`、`js/app.js`、`js/config.js`、`js/utils.js` 等处仍存在参考站品牌残留
- 主目录缺少与参考站同级的一些部署相关文件，例如 `robots.txt`、`sitemap.xml`、`favicon.ico`

### 安全与交接风险

- `git_history.log` 含敏感凭证信息
- 当前说明文档长期失真，导致“看文档理解的项目”和“实际文件夹里的项目”不是同一件事
- 素材、参考站、主源码三套来源没有被明确映射，后续很容易误删、误改、误覆盖

## 4. 接手建议顺序

建议按以下顺序接手，不要一上来直接改视觉细节。

1. 先读 `docs/00-开发者先看.md`
2. 再读 `docs/01-目录结构说明.md`
3. 再读 `docs/02-现状审计与风险清单.md`
4. 对照查看 `chocopei-website/` 与 `参考-zxwlat/`
5. 确认 `可用素材/` 里哪些素材准备接入站点
6. 按 `docs/04-开发落地路线图.md` 的顺序推进修复

## 5. 建议的工作原则

- 以 `chocopei-website/` 作为唯一主开发目录
- 以 `参考-zxwlat/` 作为对照，不直接混改
- 从 `可用素材/` 补件时，先重命名和归类，再接入
- 先修结构与配置，再做视觉升级
- 每修复一处文档失真，就同步更新 `docs/`
