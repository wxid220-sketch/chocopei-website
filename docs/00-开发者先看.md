# 开发者先看

这是一份 3 分钟接手版说明，面向第一次打开本项目文件夹的开发者。

## 1. 项目目标

当前目标是把这个工作区整理成一个可持续开发、可恢复、可交付的个人网站项目目录。  
核心不是“再写一版新站”，而是：

- 识别当前真正的主开发目录；
- 明确参考站、原始素材、历史交付来源分别是什么；
- 先修复结构和配置问题，再推进视觉与交互完善；
- 最终让后续开发者能基于同一套清晰目录继续工作。

## 2. 当前真实状态

请先接受一个事实：当前文件夹里同时存在多种来源，不能把所有内容都当作“同一个完成版本”。

### 当前可直接判断的角色分工

- `chocopei-website/`：当前主开发目录
- `参考-zxwlat/`：完整参考站
- `可用素材/`：原始素材库
- `ZXW.LAT_网站成品_交付包.zip`：历史交付/恢复来源
- `修改完善.docx`：需求来源文档

### 当前真实问题

- `chocopei-website/js/config.js` 已有损坏和拼接痕迹
- 主源码里存在资源路径与真实 `assets` 结构不一致的问题
- `404.html` 和部分脚本、样式仍保留 `ZXW.LAT` 品牌
- 旧 README 写过一些并不存在的文件，不能继续当作事实依据
- `git_history.log` 含敏感信息，属于安全风险

## 3. 先看哪些文件

建议按下面顺序看，不要跳着看。

1. 根目录 `README.md`  
   先建立对整个工作区的分工认知。

2. `docs/01-目录结构说明.md`  
   了解哪些目录是主开发区、哪些只是参考或素材来源。

3. `docs/02-现状审计与风险清单.md`  
   先知道问题，再决定怎么改。

4. `chocopei-website/index.html`  
   这是当前主站页面入口。

5. `chocopei-website/js/config.js`  
   这是当前最值得优先检查的配置断点。

6. `参考-zxwlat/` 下对应文件  
   当主开发目录出现缺件、路径疑问、行为异常时，用参考站对照。

7. `可用素材/`  
   明确哪些素材还没正式整理进站点资源目录。

## 4. 第一天该做什么

### 第一步：确认工作边界

- 把 `chocopei-website/` 认定为唯一主开发目录
- 不在 `参考-zxwlat/` 里直接开发
- 不直接把 `可用素材/` 原封不动拖进站点目录

### 第二步：做一次结构核对

至少核对以下事项：

- `index.html` 实际引用了哪些 CSS、JS、图片、音频
- `js/config.js` 中配置项是否还能与真实资源对上
- `assets/photos/` 与 `assets/thumbnails/` 的命名规则是否统一
- 当前缺哪些部署文件可从 `参考-zxwlat/` 或 ZIP 中恢复

### 第三步：处理高风险项

优先级建议如下：

1. 标记并隔离 `git_history.log` 的安全风险
2. 修复 `js/config.js`
3. 修正资源路径错位
4. 清理 `ZXW.LAT` 品牌残留
5. 统一 README 与 `docs/` 中的事实描述

### 第四步：形成第一版修复清单

第一天结束前，最好能得到这 4 个明确结论：

- 哪些文件是可继续沿用的
- 哪些文件需要从参考站或 ZIP 恢复
- 哪些素材还未正式接入主站
- 下一步应该先修结构、还是先修资源、还是先修品牌文案

## 5. 一句话建议

先把它当成“待修复的结构化网站工作区”，不要把它当成“现成成品站”。
