# 现状审计与风险清单

本文基于当前工作区可见结构和已知审计结论整理。  
目标不是追责，而是给后续开发者一个明确、可执行的风险分级视图。

## 1. 审计结论总览

当前工作区已经具备：

- 一个明确的主开发目录：`chocopei-website/`
- 一个完整的参考站：`参考-zxwlat/`
- 一组原始素材来源：`可用素材/`
- 一个历史交付/恢复来源：`ZXW.LAT_网站成品_交付包.zip`

但项目仍处于“可修复的半成品状态”，不是“结构完整、文档准确、可直接上线的交付状态”。

## 2. 阻塞开发

这类问题会直接影响后续开发者判断当前代码是否可信，建议优先处理。

### 2.1 `js/config.js` 损坏

位置：`chocopei-website/js/config.js`

现象：

- 文件内有明显拼接和继承痕迹；
- 配置内容混杂旧品牌和新项目目标；
- 部分资源路径与真实文件结构不匹配。

风险：

- 配置无法作为唯一可信来源；
- 后续开发者难以判断哪些值仍在使用；
- 页面功能、素材引用、品牌文案都会受影响。

建议：

- 先把 `config.js` 当作待重建文件；
- 对照 `index.html`、`app.js`、`参考-zxwlat/js/config.js` 重新梳理配置项；
- 只保留当前项目真实需要的字段。

### 2.2 资源路径错位

位置：主要集中在 `chocopei-website/js/config.js` 及其关联引用。

现象：

- 真实文件使用 `gallery-01.webp` 这类命名；
- 但现有配置中出现 `gallery_01.jpg` 这类并不存在的路径写法；
- 部分音频路径也表现出目录层级或命名不一致问题。

风险：

- 页面静态资源加载失败；
- 修复时难以判断是“缺文件”还是“写错路径”；
- 会进一步放大配置损坏问题。

建议：

- 基于 `chocopei-website/assets/` 做一轮实际清点；
- 统一真实命名规则；
- 修改引用时，以当前真实文件存在性为准。

### 2.3 文档与真实结构不一致

位置：旧版 `README.md` 及其衍生理解。

现象：

- 旧文档把不存在的文件写成已存在；
- 旧文档把项目说成已完成的完整站点；
- 真实目录里并没有文档中描述的全部资源和页面。

风险：

- 新接手开发者会被错误信息带偏；
- 开发排查成本上升；
- 交付判断容易失真。

建议：

- 以当前工作区实际文件为准；
- 后续每次补件或删减，都同步更新 `docs/`。

## 3. 影响交付

这类问题未必立刻阻断开发，但会明显影响上线质量、验收可信度和项目统一性。

### 3.1 品牌残留

已知位置：

- `chocopei-website/404.html`
- `chocopei-website/css/style.css`
- `chocopei-website/js/app.js`
- `chocopei-website/js/config.js`
- `chocopei-website/js/utils.js`

现象：

- 页面标题、欢迎语、内部文案仍出现 `ZXW.LAT` 或 `zxw.lat`；
- 当前项目品牌与参考站品牌混杂存在。

风险：

- 用户端看到错品牌；
- 交付时显得像未清理的套壳站；
- 后续需求判断会混淆“参考逻辑”和“当前品牌”。

建议：

- 统一列出所有品牌残留点；
- 明确当前项目品牌词、站点标题、域名展示文案；
- 一次性清理，而不是边改边漏。

### 3.2 缺文件或未补齐部署文件

对照 `参考-zxwlat/`，当前主开发目录中未看到同级的一些部署文件，例如：

- `robots.txt`
- `sitemap.xml`
- `favicon.ico`

风险：

- 搜索引擎和部署兼容性补件不完整；
- 项目看起来像“复制到一半”的状态；
- 后续发布验收时会出现基础项缺失。

建议：

- 先判断这些文件是否当前项目确实需要；
- 若需要，优先从参考站或历史交付结构中恢复，再按当前项目改名和改配置。

### 3.3 主源码目录中存在异常残留文件

位置：`chocopei-website/js$f`

现象：

- 文件名异常；
- 内容与 `app.js` 有重复痕迹；
- 职责不明确。

风险：

- 容易误以为是正式文件；
- 影响开发者对源码边界的判断；
- 若被误引用或误修改，会造成混乱。

建议：

- 先核实是否为误生成副本；
- 明确后再决定删除、归档或忽略。

## 4. 建议优化

这类问题不会立刻阻断推进，但越早整理，后续维护成本越低。

### 4.1 素材来源未形成明确映射

现象：

- `可用素材/`、`参考-zxwlat/`、`chocopei-website/assets/` 三者关系需要人工推断；
- 还没有统一说明“哪些素材已正式接入，哪些只是备选来源”。

风险：

- 易重复导入素材；
- 易覆盖已优化资源；
- 易导致不同尺寸、不同格式混用。

建议：

- 建立素材映射说明；
- 统一命名和分层规则；
- 只把最终使用的资源放入主站 `assets/`。

### 4.2 需求、实现、素材三条线未完全对齐

现象：

- 需求文档、当前源码、参考站交互并非完全一致；
- 当前实现更像“参考站改造中途版本”。

风险：

- 可能修了参考站逻辑，却没服务当前目标；
- 也可能为当前目标做了改动，却没有匹配素材。

建议：

- 每推进一步，都明确当前动作属于“恢复缺件”还是“新项目定制”；
- 避免两种动作混在一起。

### 4.3 文档失真历史需要止损

现象：

- 过去的文档已经出现“描述超前于实际文件”的情况。

风险：

- 后续如果继续这样维护，项目会越来越难交接。

建议：

- 文档只写当前可见事实和已确认计划；
- 不把“希望有的内容”写成“已经有的内容”。

## 5. 安全风险专项

### `git_history.log` 含敏感 Token

这是当前最明确的安全问题。

建议立即执行：

1. 不再外发该文件
2. 不再上传到仓库
3. 轮换相关 GitHub Token
4. 评估历史提交或共享记录中是否也包含同类敏感信息

## 6. 优先级建议

推荐处理顺序：

1. 安全风险隔离：`git_history.log`
2. 配置修复：`js/config.js`
3. 资源路径对齐
4. 品牌残留清理
5. 部署基础文件补齐
6. 素材命名与映射整理
7. 文档持续同步

## 7. 结论

当前项目最主要的问题不是“没代码”，而是“代码、素材、参考、文档之间还没有被整理成一个可信的开发工作区”。  
只要先完成结构修复和事实校准，后续开发是可以继续推进的。
