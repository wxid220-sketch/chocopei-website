# 开发落地路线图

本文给出当前项目推荐的实施顺序。  
核心原则是：先修结构，再修配置，再修资源，再做视觉与交互完善，最后做部署验收。

## 1. 路线总览

建议按以下五步推进：

1. 修结构
2. 修配置
3. 修资源
4. 做视觉与交互完善
5. 做部署验收

不要跳步。  
如果在结构和配置还不可信的情况下直接改 UI，很容易返工。

## 2. 第一步：先修结构

### 目标

把当前工作区整理成一个“开发者一看就知道哪部分该改、哪部分不能乱动”的结构。

### 输入

- `README.md`
- `docs/01-目录结构说明.md`
- `chocopei-website/`
- `参考-zxwlat/`
- `可用素材/`
- `ZXW.LAT_网站成品_交付包.zip`

### 需要完成的事

- 明确 `chocopei-website/` 是唯一主开发目录
- 明确 `参考-zxwlat/` 是对照参考，不直接混改
- 明确 `可用素材/` 是原始素材库，不直接作为线上资源目录
- 标记异常文件，例如 `chocopei-website/js$f`
- 明确哪些部署文件在主站缺失但参考站存在

### 输出

- 统一的目录职责认知
- 一份可执行的结构补件清单
- 一份需要保留、需要隔离、需要删除的文件判断清单

## 3. 第二步：再修配置

### 目标

恢复配置文件可信度，让页面入口、业务配置、资源引用之间重新建立稳定关系。

### 输入

- `chocopei-website/index.html`
- `chocopei-website/js/config.js`
- `chocopei-website/js/app.js`
- `参考-zxwlat/js/config.js`

### 需要完成的事

- 识别 `config.js` 中的损坏拼接部分
- 删除或重写无效、错误、旧品牌字段
- 根据真实资源结构重建图片、音频、文案等配置项
- 核对 `index.html` 与 `app.js` 的实际依赖关系

### 输出

- 一份可信的 `config.js`
- 一份“页面实际使用哪些配置项”的说明
- 一份待继续补齐的配置项清单

## 4. 第三步：修资源

### 目标

把当前主站引用的资源整理成可持续维护的正式资源结构。

### 输入

- `chocopei-website/assets/`
- `参考-zxwlat/assets/`
- `可用素材/`
- `docs/03-素材映射与使用建议.md`

### 需要完成的事

- 盘点 `assets/audio/`、`assets/photos/`、`assets/thumbnails/` 的真实内容
- 修正页面和配置中错误的资源路径
- 从参考站或原始素材中补齐缺少的正式资源
- 统一命名规范
- 明确哪些资源是正式上线资源，哪些只是素材库备选

### 输出

- 统一后的主站资源目录
- 一份资源命名规则
- 一份素材来源映射结果

## 5. 第四步：做视觉与交互完善

### 目标

在结构、配置、资源都可信之后，再去统一品牌和优化体验。

### 输入

- `chocopei-website/index.html`
- `chocopei-website/css/style.css`
- `chocopei-website/css/responsive.css`
- `chocopei-website/js/*.js`
- `修改完善.docx`
- `docs/05-需求文档阅读摘要.md`

### 需要完成的事

- 清理 `ZXW.LAT` 等旧品牌残留
- 统一页面标题、文案、欢迎语、OG 信息
- 优化当前页面视觉层级和交互一致性
- 结合需求摘要判断哪些参考站效果应该保留、哪些需要替换
- 确认 404 页面、导航、资源卡片、交互动效是否符合当前项目方向

### 输出

- 一套品牌一致的页面文案
- 一套可维护的页面视觉与交互状态
- 一份明确的“已完成体验项 / 待继续增强项”清单

## 6. 第五步：部署验收

### 目标

在上线前完成基础部署文件、路径、品牌、资源、文档的一次性核查。

### 输入

- `chocopei-website/` 全量主站文件
- 参考站中的部署辅助文件
- 当前交接文档目录 `docs/`

### 需要完成的事

- 检查是否需要补齐 `robots.txt`、`sitemap.xml`、`favicon.ico`
- 检查 `manifest.json`、`sw.js`、`CNAME` 是否与当前项目一致
- 检查页面是否还存在旧品牌文本
- 检查静态资源是否全部可加载
- 更新 README 和文档，确保与最终结构一致

### 输出

- 一份可验收的主站目录
- 一份上线前检查清单
- 一份与实际文件结构一致的最终交接文档

## 7. 每一步的完成标准

### 结构修复完成标准

- 接手人不再需要猜哪个目录才是主目录
- 参考站、素材库、恢复来源边界清晰

### 配置修复完成标准

- `config.js` 不再出现明显损坏和错路径
- 页面引用逻辑可以被正常追踪

### 资源修复完成标准

- 主站资源目录命名统一
- 页面实际引用的资源都真实存在

### 视觉与交互完成标准

- 不再出现旧品牌残留
- 页面语言、文案、视觉风格统一

### 部署验收完成标准

- 主站目录具备可部署性
- 文档与真实结构一致
- 高风险项已被处理或明确记录

## 8. 推荐执行方式

如果只安排一个开发者接手，建议按“结构修复 → 配置修复 → 资源修复 → 页面完善 → 验收”单线推进。  
如果多人并行，建议至少拆成以下方向：

- 一人负责结构与配置
- 一人负责资源整理与命名
- 一人负责品牌清理与页面完善

前提是三方都以 `chocopei-website/` 为唯一主开发目录。

## 9. 结论

当前项目最正确的推进方式，不是“继续堆新效果”，而是先把它整理成一个可信的项目。  
只有结构可信、配置可信、资源可信，后面的视觉和交互优化才不会反复返工。
